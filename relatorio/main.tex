\chapter{Introdução}

\section{Enquadramento}
No âmbito prático de programação e implementação de sistemas de rede, foi proposta a construção de um sistema distribuído baseado na típica arquitetura cliente-servidor. O projeto selecionado consiste no desenvolvimento  do \textbf{Jogo da Forca Multijogador}, utilizando a linguagem de programação Java e a comunicação de dados bidirecional assente em \textit{sockets} TCP.

Este tipo de arquitetura é utilizada em aplicações da atualidade, que requerem comunicação extremamente fiável e trocas de dados em tempo real. Todo o código-fonte que sustenta a implementação deste sistema pode ser consultado publicamente no repositório \textit{GitHub} através do endereço \url{https://github.com/Vascorc/Hangman.git}.


\section{Objetivos}
O objetivo principal desta proposta é criar um servidor resistente que coordene várias partidas do Jogo da Forca ao mesmo tempo, permitindo que vários jogadores participem em simultâneo. Também pretende organizar melhor a comunicação com os clientes e tornar a interface do utilizador mais clara e interativa.

Estes são os principais objetivos do programa:
\begin{enumerate}
    \item Criar a base do sistema Cliente-Servidor usando \textit{sockets} (ServerSocket e Socket em Java). 
    \item Permitir várias ligações ao mesmo tempo (entre 2 a 4 jogadores), com início automático do jogo após algum tempo de espera no lobby.
    \item Definir um protocolo de comunicação simples, baseado em mensagens de texto trocadas entre cliente e servidor.
    \item Garantir que as rondas decorrem corretamente, com tempo limite para cada jogada (\textit{timeout}), e atualizar o estado do jogo (palavra escolhida aleatoriamente de uma lista de 100).
    \item Evitar erros e conflitos no jogo, assegurando que a gestão das vidas e das jogadas é feita de forma segura e organizada.
\end{enumerate}

\section{Estruturação Arbórea do Código de Fonte}
Para organizar melhor o código e manter uma boa estrutura, o projeto foi dividido em três partes principais, cada uma com a sua função:
\begin{description}
    \item[\textit{pt.ubi.sd.forca.shared} (Repositório Comum)] Contém elementos partilhados entre cliente e servidor, como constantes, definições do protocolo e estruturas necessárias para a comunicação.
    \item[\textit{pt.ubi.sd.forca.server} (Servidor)] Responsável por gerir as ligações dos jogadores, coordenar o jogo e controlar a lógica principal, como palavras, pontuação e regras das rondas.

\end{description}

\section{Core Lógico (GameEngine e Sincronismo)}
A lógica principal do jogo é controlada por um componente chamado \textit{GameManager}, criado pelo servidor, que escolhe aleatoriamente uma palavra a partir de uma lista pré-definida com 100 opções. 
A sessão começa por aceitar ligações de jogadores, iniciando o jogo quando há pelo menos 2 participantes após um \textit{timeout} no lobby, ou imediatamente quando se atinge o máximo de 4 jogadores. Depois do jogo começar, novos jogadores são recusados através da mensagem \texttt{FULL} e obrigados a desligar.

Durante o jogo, gerido pelo \textit{GameEngine}, a jogabilidade decorre em rondas síncronas. O estado do jogo é partilhado com todos os participantes. Em cada ronda, é iniciado um cronómetro para que os jogadores submetam uma letra única ou tentem adivinhar a palavra, sendo as respostas avaliadas sem distinção entre maiúsculas e minúsculas. Respostas fora do tempo limite são penalizadas, sendo consideradas como uma submissão de texto vazio (``'').

Após todas as respostas serem recebidas, o sistema verifica os acertos, reduz o número de tentativas falhadas até um máximo definido (por exemplo, 6), e determina o fim do jogo quando a palavra é descoberta ou quando se esgotam as tentativas.

Para ilustrar o processamento de uma tentativa, o seguinte bloco de código demonstra como o motor de jogo (\textit{GameEngine}) analisa tanto caracteres individuais como tentativas completas de descobrir a palavra, reduzindo as vidas em caso de falha:

\begin{lstlisting}[caption={Processamento de palpites na classe GameEngine}]
public boolean processGuess(String guess) {
    guess = guess.toUpperCase();
    if (guess.length() == 1) {
        char letter = guess.charAt(0);
        usedLetters.add(letter);
        if (targetWord.indexOf(letter) >= 0) return true;
    } else if (guess.equals(targetWord)) {
        // Acertou a palavra toda
        for (char c : targetWord.toCharArray()) usedLetters.add(c);
        return true;
    }
    attempts--;
    return false;
}
\end{lstlisting}



\section{O Protocolo Lexical}

A comunicação entre Servidor e Cliente baseia-se em mensagens de texto, onde cada comando ocupa uma linha completa terminada por um \textit{line break}. 

\subsection{Mensagens Servidor $\rightarrow$ Cliente}
O Servidor envia comandos essenciais para controlar o jogo e informar os Clientes:

\begin{itemize}
    \item \texttt{WELCOME <id> <total>} – Cumprimento inicial, atribuindo a posição de cada jogador na sessão.
    \item \texttt{LOBBY\_UPDATE <total> <time>} – Notifica os jogadores no lobby do estado da sala (número de conexões) e decorrer do temporizador até iniciar o jogo.
    \item \texttt{START <mask> <attempts> <timeout>} – Indica o início do jogo, despachando as regras da partida (como as tentativas máximas calculadas em grupo).
    \item \texttt{ROUND <k> <mask> <attempts> <used>} – Informa sobre o novo ciclo e o estado atual.
    \item \texttt{STATE <mask> <attempts> <used>} – Atualização do estado do jogo para sincronização.
    \item \texttt{END WIN <winners> <word>} – Finaliza a partida indicando os vencedores e a palavra correta.
    \item \texttt{END LOSE <word>} – Finaliza a partida informando a palavra correta quando todos falharam.
    \item \texttt{FULL} – Rejeita jogadores que tentam entrar após a partida começar.
    \item \texttt{CANCELLED} – Notifica que o lobby foi cancelado por falta de jogadores.
    \item \texttt{PLAYER\_LEFT <id>} – Indica que um jogador se desconectou durante a partida.
\end{itemize}

\subsection{Mensagens Cliente $\rightarrow$ Servidor}
O Cliente envia apenas requisições ativas, como:

\begin{itemize}
    \item \texttt{GUESS <text>} – Submissão de uma letra ou da palavra completa.
\end{itemize}

\section{Sincronização e Exclusão Mútua}
Para garantir a integridade dos dados, foi utilizado o mecanismo de \textit{monitor} do Java através da palavra-chave \texttt{synchronized(this)}. Isto permitiu definir secções críticas no \textit{GameManager}, assegurando que, por exemplo, dois jogadores não conseguem ser adicionados à mesma posição da lista simultaneamente ou que o processamento do fim da ronda só ocorre após a receção de todas as jogadas.

O trecho de código abaixo demonstra a aplicação do \texttt{synchronized(this)} na adição de jogadores, na classe \textit{GameManager}, avaliando e atualizando a capacidade de rede de forma totalmente atómica face a pedidos paralelos:

\begin{lstlisting}[caption={Exclusão Mútua ao adicionar jogadores}]
public void addPlayer(ClientHandler player) {
    synchronized(this) {
        if (players.size() >= 4 || gameStarted) {
            player.sendMessage(Protocol.FULL);
            player.closeConnection();
            return;
        }

        players.add(player);
        int id = players.size() - 1;
        player.setPlayerId(id);

        player.sendMessage(Protocol.WELCOME + " " + id + " " + players.size());
        checkLobbyStatus();
    }
}
\end{lstlisting}

\section{Tratamento de Erros e Tolerância a Falhas}
O servidor foi desenhado para ser capaz de resistir a desconexões inesperadas. Caso um \textit{socket} seja fechado abruptamente, a \textit{thread} dedicada a esse cliente deteta a \textit{IOException}, encerra os recursos localmente e o \textit{GameManager} é notificado para remover o jogador da sessão ativa ou declarar a sua desistência.

Este comportamento foi implementado na função \texttt{playerDisconnected}, acionada de forma assíncrona assim que falhas de rede são apanhadas e processadas:

\begin{lstlisting}[caption={Tratamento de desconexões do servidor para com os jogadores}]
public synchronized void playerDisconnected(ClientHandler player) {
    players.remove(player);
    if (gameStarted) {
        // Avisa os restantes que um jogador saiu e termina o jogo
        broadcast(Protocol.PLAYER_LEFT + " " + player.getPlayerId());
        stopGame();
    } else if (players.isEmpty()) {
        // Cancela o timer do lobby se não houver mais jogadores para garantir poupança do CPU
        if (lobbyTimer != null) {
            lobbyTimer.cancel();
            lobbyTimer = null;
        }
    }
}
\end{lstlisting}

\section{Pequenas Funcionalidades}

\subsection{Contagem Regressiva Dinâmica em Consola (Códigos ANSI)}
Uma das limitações técnicas inerentes às aplicações de consola que bloqueiam a \textit{thread} principal a aguardar a leitura do teclado (\texttt{scanner.nextLine()}) é a incapacidade de lidar com atualizações fluidas na interface. A tentativa de mostrar um relógio tradicional com contagem visível a descer obrigaria a que as letras ativamente digitadas pelo jogador fossem intercetadas ou engolidas.

Para solucionar de forma elegante este desafio visual imposto, aplicámos uma abordagem assente em formatações de \textbf{ANSI escape sequences}.

A lógica opera lançando uma \textit{Thread daemon} sempre que se exige uma jogada. A cada decréscimo intervalar atómico de um segundo, utilizam-se estas tags de controlo para manipular a janela (matriz Y) da seguinte forma:
\begin{enumerate}
    \item \texttt{\textbackslash033[s} guarda na memória da consola a posição exata das coordenadas em que o curso digitável se encontra suspenso, sem imprimir nada de novo.
    \item \texttt{\textbackslash033[<N>A} movimenta temporariamente e verticalmente o indicador de ecrã para cima num espaço rígido correspondente a \textbf{N} linhas, ou seja, retrocedendo as linhas que desenharam o "Boneco" e restantes textos.
    \item Chegando ao ponto calculado pelo algoritmo, sobrepõe toda a linha para uma nova que contém o cronómetro subtraído. Dispara adicionalmente um comando de limpeza limiar (\texttt{\textbackslash033[K}).
    \item Imediatamente, e sem que a operação seja percetível à cadência de imagem dos píxeis normais, a tag de fecho \texttt{\textbackslash033[u} força a que o cursor seja reposicionado no fundo para que possa continuar ativamente a processar as teclas em ciclo de \textit{buffer} do \textit{Scanner}.
\end{enumerate}

Na porção técnica da classe de gráfica (\texttt{ConsoleUI.java}), todo este comportamento em background consome um núcleo temporal com a seguinte anatomia prototipada:

\begin{lstlisting}[caption={Rotina Assíncrona de Input com Relógio ANSI em Segundo Plano}]
public static String readGuess(Scanner scanner) {
    waitingForInput = true;
    
    Thread timer = new Thread(() -> {
        int time = 30; // 30 segundos (Tempo Limite)
        while (waitingForInput && time > 0) {
            try {
                Thread.sleep(1000);
                time--;
                if (waitingForInput) {
                    String timeStr = String.format("%02d", time);
                    // \033[s guarda cursor, \033[A navega p/ Norte e reescreve 
                    System.out.print("\033[s"); 
                    System.out.print("\033[" + linesToMoveUp + "A");
                    System.out.print("\r   RONDA " + currentRound + "   |    Tempo: " + timeStr + "s \033[K");
                    System.out.print("\033[u"); // \033[u liberta cursor para o Sul
                    System.out.flush();
                }
            } catch (InterruptedException e) { break; }
        }
    });
    timer.setDaemon(true);
    timer.start();

    String result = null;
    if (scanner.hasNextLine()) {
        result = scanner.nextLine().trim();
    }
    
    waitingForInput = false;
    timer.interrupt();
    return result;
}
\end{lstlisting}

Devido à melhoria visual proporcionada por este mecanismo durante o decorrer das rondas, a mesma ideia foi aplicada à contagem regressiva na fase inicial (\textbf{Lobby}). Assim, em vez de o terminal mostrar repetidamente mensagens como "Aguardar jogadores..." sempre que o tempo diminui ou entra um novo jogador, toda a informação é atualizada no mesmo local de forma assíncrona, reagindo a mensagens de rede do tipo \texttt{LOBBY\_UPDATE}.

\subsection{Renderização Gráfica do Boneco}
A representação visual do jogo é feita através do clássico boneco da forca. Como as consolas de texto não permitem gráficos avançados, foi utilizada \textit{Arte ASCII} para contornar essa limitação.

A lógica de desenho baseia-se no número de erros acumulados pelos jogadores (ou seja, $6 - \textit{tentativas restantes}$). Dependendo desse valor, partes do boneco vão sendo mostradas.

Para isso, são usadas condições (por exemplo, operadores ternários) que decidem quais caracteres devem aparecer, como a cabeça \texttt{'O'} ou os braços \texttt{'/'}. Caso o número de erros ainda não seja suficiente, esses elementos são substituídos por espaços em branco.

O seguinte bloco na \texttt{ConsoleUI} demonstra este funcionamento:

\begin{lstlisting}[caption={Rotina dinâmica de Renderização do Boneco em ASCII}]
public static void drawHangman(int attemptsLeft) {
    // Reduzimos as partes penduradas proporcionalmente a 6 fases
    int erroCount = 6 - (int) Math.ceil((double) attemptsLeft * 6 / Math.max(maxAttempts, 1));

    String head = erroCount >= 1 ? "O" : " ";
    String body = erroCount >= 2 ? "|" : " ";
    String lArm = erroCount >= 3 ? "/" : " ";
    String rArm = erroCount >= 4 ? "\\" : " ";
    String lLeg = erroCount >= 5 ? "/" : " ";
    String rLeg = erroCount >= 6 ? "\\" : " ";

    System.out.println("  ┌───┐");
    System.out.println("  │   " + head);
    System.out.println("  │  " + lArm + body + rArm);
    System.out.println("  │  " + lLeg + " " + rLeg);
    System.out.println("  │");
    System.out.println("──┴──");
}
\end{lstlisting}

\subsection{Apresentação de Letras Tentadas Anteriormente}
A gestão do rol de perspetivas utilizadas (o inventário de letras submetidas falhadas ou já preenchidas) reveste-se de grande utilidade para prevenir castigos acidentais na alocação de letras erróneas ativas, garantindo visibilidade total na área de jogo.

Sob a ótica operacional distribuída da arquitetura implementada, os Clientes retêm o estado de forma leviana (\textit{Stateless}). Todas as lógicas analíticas sobre as letras são processadas em \textit{Cache} pelo sistema de controlo primário associado ao objeto \texttt{GameEngine} situado no Servidor Lógico e, em última instância, unificadas num vetor textual partilhado em rede pelo protocolo nativo \texttt{ROUND} e \texttt{STATE}. 
Assim que o cliente processa e captura a carga iterativa empacotada no tráfego, o mesmo apenas projeta o resultado puramente cru, injetado submetido de inspeção pela camada visual, indicando primeiro a validade material na \textit{String} antes de desenhar a linha final com o respetivo limitador gráfico.

\chapter{Conclusões e Trabalho Futuro}

\section{Conclusão}
O desenvolvimento deste projeto permitiu consolidar os conhecimentos sobre comunicação em rede utilizando Sockets TCP e a gestão de concorrência em Java. A arquitetura cliente-servidor implementada mostrou-se robusta para o número de jogadores definido, garantindo que o protocolo de texto é seguido de forma rigorosa e síncrona. A separação de responsabilidades entre o motor de jogo e a gestão de comunicações facilitou a manutenção e o teste do sistema.

\section{Trabalho Futuro}
Para versões futuras deste sistema, poderiam ser implementadas as seguintes melhorias:
\begin{itemize}
    \item \textbf{Interface Gráfica (GUI):} Substituir a consola por uma interface JavaFX para uma melhor experiência de utilizador.
    \item \textbf{Persistência de Dados:} Implementar uma base de dados para guardar um histórico de vitórias e recordes dos jogadores.
\end{itemize}
